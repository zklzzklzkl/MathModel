# 赛题预审与上下文固化

本 skill 是数学建模工作流的前置阶段。它不直接建模、不写代码、不写论文，而是把赛题信息变成后续阶段可以反复读取的项目档案，降低长上下文丢失、题面误读和后期数值漂移风险。

## 参考

执行前读取：

- `../_references/workflow_state_contract.md`
- `../_references/math_modeling_norms.md` 中的赛题理解、假设、题型防错、论文验收相关部分

## 必须产出

在当前工作目录创建或更新：

- `PROBLEM_BRIEF.md`
- `DATA_AUDIT.md`
- `WORKFLOW_STATE.md`
- `reports/TRIAGE_REPORT.md`
- `todo.md` 中对应预审任务状态

如果 `reports/` 不存在，先创建。若后续阶段需要更多目录，也可以在本阶段创建空目录骨架。

## 工作流程

### 1. 题面与附件清点

读取题面、附件列表和用户给出的比赛背景。记录：

- 题目来源和比赛类型
- 原始题面文件或文本位置
- 附件文件名、格式、大小、可能用途
- 是否存在扫描件、图片公式、单位、表格字段或符号容易误读的风险

不要在此阶段擅自改写题意。对不确定内容写入“待确认”。

### 2. 子问题识别

只把题面中明确编号的顶层问题识别为子问题，例如“问题一/二/三”或 “Problem 1/2/3”。不要把背景说明、附件说明、提交要求、小问提示误判成独立子问题。

在 `PROBLEM_BRIEF.md` 中写明：

- 子问题数量
- 每个子问题的输入、输出、目标、约束、评价指标
- 子问题之间的依赖关系
- 每个子问题预计需要的数据和图表证据

### 3. 可行性与路线预审

从竞赛角度做可行性评估：

- 数据是否足以回答每个子问题
- 题型属于优化、预测、评价、分类、机理、图论、仿真或混合题
- 哪些模型是基准路线、主路线、备用路线
- 代码实现风险和论文表达风险
- 如果是多题选择场景，给出选题建议和换题条件

### 4. 时间预算与降级策略

在 `WORKFLOW_STATE.md` 中记录时间盒：

- 读题与预审
- 建模报告
- 代码求解
- 图表与流程图
- 论文写作
- 最终验收

同时写出降级策略：当时间不足、数据不可用、模型求解失败或图表质量不达标时，如何切换到更稳的方案。

### 5. 后续阶段交接

`reports/TRIAGE_REPORT.md` 必须包含：

```markdown
# 赛题预审报告

## 结论
GO / CONDITIONAL_GO / STOP

## 题面与附件清点

## 子问题识别

## 可行性评估

## 候选建模路线

## 数据风险

## 时间预算

## 需要人工确认的问题
```

只有当结论为 `GO` 或 `CONDITIONAL_GO` 时，才进入 `2analysis-modeling`。若为 `STOP`，说明必须补充题面、附件或用户决策。

## 质量要求

- 不编造附件字段和数据含义。
- 不提前锁死唯一模型路线。
- 不把题面提交要求误当成建模目标。
- 任何不确定点必须落到 `reports/TRIAGE_REPORT.md` 的“需要人工确认的问题”。
